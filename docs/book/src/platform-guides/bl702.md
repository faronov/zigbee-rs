# BL702

Bouffalo Lab's BL702 is a RISC-V SoC with built-in IEEE 802.15.4 and BLE 5.0
radio. The current zigbee-rs BL702 backend is an experimental FFI scaffold
around Bouffalo's binary-only radio libraries. It is compile-tested with
no-op stubs, not hardware-proven, and not pure Rust.

## Hardware Overview

| Spec | Value |
|------|-------|
| **Core** | RISC-V 32-bit (RV32IMAF), 144 MHz |
| **Flash** | 512 KB (XIP) |
| **SRAM** | 112 KiB main XIP SRAM window plus 4 KiB HBN RAM in the vendor linker |
| **Radio** | BLE 5.0 + IEEE 802.15.4 |
| **Target** | `riscv32imac` for stubs; `riscv32imafc` required by vendor ABI |
| **I/O** | UART ×2, SPI, I2C, ADC, DAC, USB 2.0 FS |

### Common Modules and Boards

- **XT-ZB1** — Zigbee module based on BL702
- **DT-BL10** — compact BL702 breakout
- **Pine64 Pinenut** — BL602/BL702 module
- **BL706 IoT DevBoard (Sipeed)** — devkit with USB and headers

### Memory Map

The vendor Zigbee linker splits the main SRAM window around a reserved 1 KiB:

```text
FLASH : ORIGIN = 0x23000000, LENGTH = 512K
RAM1  : ORIGIN = 0x42014000, LENGTH = 80K
RSVD  : ORIGIN = 0x42028000, LENGTH = 1K
RAM2  : ORIGIN = 0x42028400, LENGTH = 31K
HBN   : ORIGIN = 0x40010000, LENGTH = 4K
```

The compile-only Rust linker now keeps data/heap in RAM1 and the stack in RAM2,
so it cannot allocate over the reserved range. RF calibration also requires
its private data to finish below `0x42024000` and temporarily preserves the
1 KiB sample window beginning there.

### Pure-Rust Feasibility

The radio archives are not stripped: symbols and DWARF expose enough structure
to recover the M154 FIFO/IRQ/register paths and the RF reset and calibration
algorithms. The channel synthesizer word is exactly `frequency_mhz * 4096`;
channel 15 therefore uses `0x00979000`.

Cold boot still requires live ACAL, KCAL, ROSCAL, and RCCAL. The vendor restore
path is for retained calibration state and does not justify a generic
precomputed snapshot. A pure-Rust implementation is feasible, but remains
compile-only until startup, CLIC dispatch, live calibration, TX/RX, image
packaging, and hardware behavior are proven.

## Prerequisites

### Rust Toolchain

```bash
rustup default nightly
rustup update nightly

# Add the RISC-V target
rustup target add riscv32imac-unknown-none-elf

# rust-src for build-std
rustup component add rust-src
```

### Vendor Libraries

The SDK provides the radio implementation only as pre-compiled archives:

- **`liblmac154.a`** — 802.15.4 MAC/PHY library
- **`libbl702_rf.a`** — RF calibration and configuration

They use the `rv32imfc/ilp32f` hard-float ABI. An ABI-correct Rust experiment
must use `riscv32imafc-unknown-none-elf`. Removing the ELF float-ABI flag is
not valid because it does not convert calling conventions.

## Building

### With Stubs (CI mode — no vendor SDK)

```bash
cd examples/bl702-sensor
cargo build --release --features stubs -Z build-std=core,alloc
```

The `stubs` feature provides no-op implementations of all FFI symbols. It
checks compilation only; the resulting ELF must not be flashed.

### Vendor ABI Experiment

```bash
BL_IOT_SDK_DIR=/path/to/bl_iot_sdk \
  cargo build --release --target riscv32imafc-unknown-none-elf \
  -Z build-std=core,alloc
```

This is not currently a supported firmware build. Startup, CLIC trap dispatch,
RF prerequisites, vendor dependencies, boot-header generation, and hardware
behavior remain unproven.

### CI Build Command

From `.github/workflows/ci.yml`:

```bash
cd examples/bl702-sensor
cargo build --release --features stubs -Z build-std=core,alloc
```

CI intentionally publishes no BL702 firmware artifact because the linked radio
symbols are stubs.

### Build Script (`build.rs`)

The `build.rs` handles vendor library discovery:

```rust
// Skip vendor libs when `stubs` feature is active (CI mode)
let use_stubs = env::var("CARGO_FEATURE_STUBS").is_ok();
if use_stubs { return; }

// Priority: BL_IOT_SDK_DIR → LMAC154_LIB_DIR → vendor_libs/
```

### `.cargo/config.toml`

```toml
[build]
target = "riscv32imac-unknown-none-elf"

[unstable]
build-std = ["core", "alloc"]
```

## Hardware Status

Do not flash the stub build. A flashable image requires a verified BL702 boot
header, partition/layout checks, startup and CLIC dispatch, and a real radio
implementation.

## MAC Backend Notes

The BL702 MAC backend lives in `zigbee-mac/src/bl702/`:

```
zigbee-mac/src/bl702/
├── mod.rs      # Bl702Mac struct, MacDriver trait impl, PIB management
└── driver.rs   # Bl702Driver — FFI bindings to lmac154.a
```

### Feature Flag

```toml
zigbee-mac = { features = ["bl702"] }
```

### Architecture

```
MacDriver trait methods
       │
       ▼
Bl702Mac (mod.rs)
  ├── PIB state (addresses, channel, config)
  ├── Frame construction (beacon req, assoc req, data)
  └── Bl702Driver (driver.rs)
         ├── FFI → liblmac154.a (Bouffalo C library)
         │     ├── lmac154_setChannel / setPanId / setShortAddr
         │     ├── lmac154_triggerTx / enableRx / readRxData
         │     └── lmac154_runCCA / getRSSI / getLQI
         ├── TX completion: lmac154_txDoneEvent callback → TX_SIGNAL
         └── RX completion: lmac154_rxDoneEvent callback → RX_SIGNAL
```

### FFI Callbacks

The vendor library calls back into Rust through these functions:

```rust
#[no_mangle]
extern "C" fn lmac154_txDoneEvent(/* ... */) {
    TX_SIGNAL.signal(tx_status);
}

#[no_mangle]
extern "C" fn lmac154_rxDoneEvent(/* ... */) {
    // Copy RX data to static buffer
    RX_SIGNAL.signal(());
}
```

### Interrupt Registration

The vendor example registers the M154 handler returned by
`lmac154_getInterruptHandler()`. The Rust scaffold does not yet provide a
proven machine-trap/CLIC dispatch path:

```rust
// bl_irq_register(M154_IRQn, lmac154_getInterruptHandler());
// bl_irq_enable(M154_IRQn);
```

### Radio Features

- Hardware CRC generation and checking
- Configurable TX power: 0 dBm to +14 dBm
- RSSI / LQI measurement
- Hardware auto-ACK with configurable retransmission
- Hardware address filtering (PAN ID, short addr, long addr)
- CSMA-CA support
- AES-128 CCM hardware acceleration

## Example Walkthrough

The `bl702-sensor` example implements a Zigbee 3.0 temperature & humidity
end device with button control and UART logging.

### Custom Embassy Time Driver

Since there is no `embassy-bl702` HAL yet, the example provides a minimal
Embassy time driver using the BL702 TIMER_CH0:

```rust
// 32-bit match-compare timer at 1 MHz (FCLK/32 from 32 MHz)
const TIMER_BASE: usize = 0x4000_A500;

pub fn init() {
    // Prescaler: FCLK(32 MHz) / 32 = 1 MHz tick
    write_volatile(TCCR, 0x1F);
    // Free-running mode
    write_volatile(TMR_0, 0xFFFF_FFFF);
    // Enable counter
    write_volatile(TCER, 0x01);
}
```

### UART Logging

Debug output goes through the BL702 UART peripheral:

```rust
impl log::Log for Bl702Logger {
    fn log(&self, record: &log::Record) {
        let mut w = UartWriter;
        write!(w, "[{}] {}\r\n", record.level(), record.args());
    }
}
```

### GPIO Button

Direct register-based GPIO for the join/leave button:

```rust
const BUTTON_PIN: u8 = 8;  // GPIO8 on most BL702 modules
gpio::configure_input_pullup(BUTTON_PIN);
```

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| Linker error: undefined `lmac154_*` | Vendor libraries not linked | Set `BL_IOT_SDK_DIR` or use `--features stubs` |
| Float ABI mismatch | Vendor `.a` uses `ilp32f` | Use an ABI-correct `riscv32imafc` build; never strip the flag |
| `blflash` can't connect | Not in boot mode | Hold BOOT pin low during reset |
| No UART output | Wrong UART pins or baud | Check board schematic; default 115200 baud |
| Timer not working | TIMER_CH0 not initialized | Ensure `time_driver::init()` runs before Embassy |
| Build fails without `stubs` | Missing vendor lib env vars | Set `LMAC154_LIB_DIR` or `BL_IOT_SDK_DIR` |
